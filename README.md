## Callie: Chosen Scenario and Four Additional Elements

Our team has chosen the third-person game scenario. We are planning to make a single-level horror hotel game with a clear starting area, a final goal area, and at least five rooms of different sizes.

The four additional elements we chose are:

- Player Power-Up Pickups: Health kits, flashlight batteries, and other helpful items that can be used for survival or puzzles.
- Enemies (Stationary): Haunted hotel objects like suitcases, portraits, lamps, and the front desk bell that can damage or affect the player.
- Enemies (Moving): Haunted bellhops, hotel staff, or guests that move around the hotel and can chase or attack the player.
- Enemies (Bosses): A hotel manager boss who acts as the final major enemy encounter.

## Kay: Developmental Goals
- For our Alpha build, we will focus on making sure the third-person player works, the hotel level is blocked out, the starting area and goal area are in, one pickup works, one enemy or haunted object works, and having at least a basic HUD implemented.
- For our Beta build, our goal is to have the level playable from start to finish. This means having most of the pickups, enemies, boss/progression mechanics, UI, and hotel visuals working. This will be the version where we test everything, fix major issues, and figure out what still needs polish before the final release.

### Testing During the Development Cycle

Our team will test the game throughout the development cycle instead of waiting until the final release. Since our game is a third-person horror hotel level, testing will focus on making sure the player can move through the level, interact with objects, encounter enemies, understand the objective, and reach the final goal.

Our current core gameplay loop is:

Player starts in the hotel, explores the level to unlock room doors, interacts with or collects useful items, encounters threats, and reaches the final goal to escape alive.

Testing will focus on making sure this core loop works before we add too many extra features or polish. This will help us avoid overscoping and make sure we have a playable Alpha, Beta, and Final release.

### Play Test: Preproduction Testing

During the preproduction/play test stage, our team will test the smallest playable version of the game. The goal is to make sure the foundation works before we spend too much time adding final visuals, extra mechanics, or polish.

Items to test during this stage include:

- Player movement and camera
- Player spawn/start location
- Basic hotel level blockout
- Room layout and player flow
- Basic interaction with at least one pickup or haunted object
- Early UI/HUD elements
- Whether the player can reach a goal area
- Whether the level structure supports the horror hotel theme

The purpose of this stage is to confirm that the basic version of the game is playable and that the player can understand where to go and what to do.

### Demo Testing

Before the demo stage, our team will test the game as a more complete playable experience. The goal is to make sure the level can be played from beginning to end and that the main gameplay elements are working.

Items to test during this stage include:

- Full level progression from start to finish
- At least five rooms of varying sizes
- Power-up pickups, such as health kits or flashlight batteries
- Stationary enemies, such as haunted hotel objects
- Moving enemies, such as haunted hotel staff or guests
- Hotel manager boss behavior
- UI/HUD clarity
- Objective prompts and interaction prompts
- Lighting, props, textures, and horror hotel theme
- Major bugs that could interrupt gameplay

### Code Release Testing

Before the final code release, our team will use the test plan to check the full game and confirm that it meets the project requirements. This stage will focus on bug fixes, polish, and making sure the final version is fully playable.

Items to test during this stage include:

- Full playthrough from start to finish
- All required level elements
- All four selected additional elements
- Final goal or win condition
- UI/HUD functionality
- Textures and visual consistency
- Major bugs fixed
- 5–10 minute playtime goal
- Overall level functionality and stability

### Updating the Test Plan

The test plan will be updated as the game changes. If the team adds, removes, simplifies, or changes a feature, the testing checklist will be updated to match the current version of the game.

The test plan may be updated when:

- A new mechanic is added
- A feature is removed or simplified
- The level layout changes
- A new pickup, enemy, boss, or UI element is added
- A bug reveals that more testing is needed
- The design document or project scope changes

This will help make sure the test plan stays connected to the actual version of the game we are building.

### Bug Reporting

Bugs will be reported through Discord and may also be tracked through GitHub Issues if needed. When reporting a bug, team members should include enough information for the rest of the team to understand and recreate the issue.

Bug reports should include:

- Bug title
- Area or feature affected
- Steps to reproduce the bug
- Expected result
- Actual result
- Severity level
- Screenshot or video, if helpful

Severity levels:

- Low: Minor visual or polish issue
- Medium: Noticeable issue, but the game is still playable
- High: Major issue that affects gameplay
- Blocker: Issue prevents progress or stops the game from being completed

### Tracking Bugs Over Time

Bugs and fixes will be tracked through Discord updates, GitHub commits, and GitHub Issues if the team decides to use them. When a bug is fixed, the team member who fixed it should post an update and commit the change to the repository with a clear commit message.

Example commit messages:

- Fix player pickup interaction
- Update haunted object damage
- Fix hotel room collision issue
- Update HUD health display
- Adjust boss enemy behavior

During weekly check-ins, the team will review open bugs and decide which issues need to be fixed first. Higher priority will be given to bugs that block gameplay, prevent level completion, or affect the Alpha, Beta, or Final release requirements.
